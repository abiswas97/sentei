## Design Context

### Users
Developer power users managing multiple git worktrees in bare repository setups. They use the terminal daily, expect keyboard-driven workflows, and value tools that are fast, safe, and informative. The primary job: manage the full worktree lifecycle (create, setup, remove, cleanup) without memorizing scripts or flags.

### Brand Personality
Friendly, polished, capable. Like a well-made tool that feels good to use — not flashy, but clearly crafted with care. Think the Charm ecosystem (gh, gum, soft-serve) rather than raw ncurses.

### Aesthetic Direction
- **Tone:** Approachable and polished. Good use of whitespace, clear hierarchy, thoughtful color.
- **References:** gh CLI, lazygit (information density), Charm tools (visual polish).
- **Anti-references:** Dense walls of unformatted text. Overly colorful / rainbow terminal output.
- **Theme:** Adaptive — detect terminal background and adjust colors for both dark and light terminals.
- **Current palette:** Purple accent (ANSI 62), semantic status colors (green 42 = clean, yellow 214 = dirty/warning, red 196 = error/untracked), gray 241 for dim/secondary text, white 15 for emphasis. Rounded borders for dialogs.

### Visual Evolution
As sentei grows from a cleanup tool to a full worktree manager, the visual identity should evolve with scope. New views (creation flows, repo operations) can introduce richer patterns — step indicators, section headers, progress sequences — while maintaining the core color vocabulary and Charm-style polish.

### Design Principles
1. **Information hierarchy first** — Most important data (branch name, status, age) should be instantly scannable. Secondary details (commit subjects, paths) are available but don't compete.
2. **Color means something** — Every color carries semantic meaning (status, emphasis, warning). Never use color purely for decoration.
3. **Keyboard-native** — Every action is reachable via keyboard. Vim-style (j/k) plus standard keys (arrows, space, enter). Discoverable via status bar hints.
4. **Safe by default** — Destructive actions require confirmation. Warnings are visually distinct (yellow/red). Protected items are clearly marked and non-selectable.
5. **Progressive disclosure** — Show the essential view first, let users drill into details. Don't front-load every piece of information.

## Component Patterns

### View Chrome
Every full-screen view renders the same frame, via the pure helpers in `internal/tui/chrome.go`:
- **Title:** `viewTitle("View Name")` → `  sentei ─ View Name`, bold white (15). No backgrounds, no badges.
- **Subtitle (optional):** one dim or accent context line under the title (repo name, branch arrow).
- **Separators:** `viewSeparator(width)` → dim (241) `┄` rule, 2-space pad, width-4 long. One above content, one below.
- **Key hints:** `viewKeyHints(KeyHint{key, action}...)` → dim, joined with ` · `. Always the last line. Never use `•`.

### Progress Views
All long-running flows render through `ProgressLayout` (`internal/tui/progress_layout.go`) fed `phaseDisplay` data:
- Phase header: `  <indicator> <Name>  <done>/<total>  <pct>%`, indicator left of the name.
- Steps indent 4 spaces under their phase; completed phases collapse to one line unless they contain failures.
- A phase with no discovered work (`total == 0`) is `pending`, never 100%.
- Overall bar: 20 cells, accent (62) fill, dim (241) track, percentage in default foreground. Clamped to 0-100.
- Large step lists window to terminal height: failed and active steps always visible, then `WindowCompletedTrail` recent completions and `WindowPendingLead` upcoming, with a stat line (`● N done · ◐ N active · · N pending  showing X of Y`).

### Status Indicators
| Char | Meaning | Color |
|------|---------|-------|
| `●` | done | green 42 |
| `◐` | active | accent 62 |
| `·` | pending | gray 241 |
| `✗` | failed | red 196 |
| `⚠` | warning | orange 214 |

`●` never means pending. Pending is always `·`.

### Text Overflow
Paths, branch names, and error text that can exceed the width go through `truncateWithEllipsis` — content never hard-clips at the terminal edge.

### Timing
Progress views hold for `minProgressDuration` (`WithMinProgressDuration`, 1.5s in playground) via `holdOrAdvance` before advancing, so fast operations stay legible. There is no other timing mechanism.

### Key Mapping
All bindings live in `internal/tui/keys.go` and are referenced, never redefined locally. `?` is contextual info for the current view; `F1` is reserved for global help (portal change).

### Dialogs
Confirmations are full-screen steps using the standard chrome, not bordered boxes. The only bordered overlay is the integration info card (`styleInfoCard`), which the DetailPortal component will absorb.
