  Now, the debate on confirmation in menu flow:

  The case for skipping confirmation in menu flow:
  - User already made a deliberate choice: navigated to menu → selected "Cleanup
   & exit" → that's already two actions expressing intent
  - Confirmation adds friction to the common case without adding safety —
  cleanup in safe mode is non-destructive (prune stale refs, remove gone
  branches)
  - The CLI command echo is useful for discoverability, but there are better
  places for it (summary view, --help)

  The case for keeping confirmation everywhere:
  - Consistency — the mental model is "confirmation before any mutation." If
  some flows skip it and others don't, users can't predict the behavior
  - Safe mode is non-destructive, but aggressive mode deletes branches. If we
  skip confirmation, we need to be sure the mode is always what the user expects
  - The confirmation screen shows the effective options — mode, dry-run status.
  Even from the menu, this gives the user a moment to verify

  My take: I think the answer depends on whether the menu itself shows the mode.
   Right now:

  CURRENT MENU:                         WHAT IF MENU SHOWED MODE:
  ┌────────────────────────┐           ┌──────────────────────────────┐
  │  > Create worktree     │           │  > Create worktree           │
  │    Remove worktrees    │           │    Remove worktrees          │
  │    Manage integrations │           │    Manage integrations       │
  │    Cleanup & exit      │ ← opaque  │    Cleanup & exit (safe)     │ ←
  transparent
  └────────────────────────┘           └──────────────────────────────┘

  If the menu item already tells you the mode, the confirmation is truly
  redundant. If it doesn't, the confirmation is the only place you see what's
  about to happen.

  On --yes / --non-interactive: This is the right escape hatch for CI/scripting.
   The pattern I'd recommend:

  - sentei cleanup — interactive, shows confirmation
  - sentei cleanup --yes — skip confirmation, run immediately
  - From TUI menu — skip confirmation (user already chose interactively), go
  straight to progress/result
  - The CLI command echo moves to the summary view: "Equivalent: sentei cleanup 
  --mode=safe"

  This means confirmation is a CLI-only concept — it exists to verify flags you
  passed on the command line. In the TUI, you're already in an interactive
  context where every action is deliberate.

  The --yes flag also gives you the non-interactive path for scripting (sentei 
  cleanup --mode=aggressive --yes).

  And should --yes also apply to other CLI subcommands that have confirmations

  Section 2: Unified Progress Layout

  The core rendering pattern shared by all 4 progress views. This is a rendering
   function, not a component with its own Model — it takes data and returns a
  string.

  // internal/tui/chrome.go (or progress_layout.go)

  type ProgressPhase struct {
      Name    string
      Steps   []ProgressStep
      Done    int
      Total   int
      Failed  int
  }

  type ProgressStep struct {
      Name   string
      Status StepStatus  // done, active, failed, pending
      Error  error
  }

  type ProgressLayout struct {
      Title      string
      Subtitle   string          // optional — e.g., "feature/foo → from main"
      Phases     []ProgressPhase
      Width      int
      Height     int             // for adaptive windowing
  }

  func (p ProgressLayout) View() string

  The View() renders this standard layout:

    sentei ─ <Title>

    <Subtitle>                        ← only if set

    ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄

    <Phase 1>                100% ●
                                      ← steps hidden when complete (unless
  failed)
    <Phase 2>              3/10  30%
      ● step-done
      ◐ step-active
      · step-pending
      ● 4 done · ◐ 1 active · · 5 pending  showing 3 of 10
                                      ← stat line only when windowed
    <Phase 3>              pending

    ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄

    ████████░░░░░░░░░░░░ 30%         ← overall progress bar

    ctrl+c quit

  Windowing logic (inside View()):
  1. Compute fixedLines = title + subtitle + separators + bar + hints + other
  phases
  2. availableForSteps = p.Height - fixedLines
  3. If phase.Total <= availableForSteps: show all steps, no stat line
  4. If phase.Total > availableForSteps:
    - Always show: all failed (✗) + all active (◐)
    - Fill remaining: last N completed (●) + next M pending (·)
    - Add stat line: ● X done · ◐ Y active · · Z pending  showing W of Total
    - Reserve 1 line for stat line in the budget

  Overall progress bar computes across all phases: totalDone / totalSteps.

  Each existing view becomes a thin adapter that builds a ProgressLayout from
  its domain-specific events and calls .View(). For example, viewProgress()
  (removal) maps deletionStatuses to ProgressPhase/ProgressStep,
  viewCreateProgress() maps creator.Event to the same types.

  Section 3: Adaptive Windowing

  A pure function that selects which steps to display given a budget:

  // internal/tui/window.go

  type WindowResult struct {
      Visible []ProgressStep  // steps to render
      Stats   WindowStats     // for the stat line
      Windowed bool           // whether windowing is active
  }

  type WindowStats struct {
      Done    int
      Active  int
      Pending int
      Failed  int
      Total   int
      Showing int
  }

  // WindowSteps selects which steps to display given available lines.
  // Priority: failed (always) > active (always) > last N done > next M pending.
  func WindowSteps(steps []ProgressStep, availableLines int) WindowResult

  Rules:
  1. Failed items are always visible (pinned) — you need to see what broke
  2. Active items are always visible — that's the current action
  3. Remaining budget fills with: last completed (most recent first) + next
  pending
  4. If len(steps) <= availableLines: return all, Windowed = false
  5. Stat line consumes 1 line from budget when Windowed = true

  func viewStatLine(stats WindowStats) string
  // renders: "    ● 10 done · ◐ 3 active · · 17 pending  showing 6 of 30"

  This is pure logic — no Model, no TUI state. Testable with table-driven tests

  - Active phases expand to show steps
  - Overall progress bar below bottom separator: `████░░░░ XX%`

  | · | pending | gray 241 | Not started |
  | ✗ | failed | red 196 | Error/failure |

  When item lists are windowed: `● N done · ◐ N active · · N pending  showing X
