## [P0] (cleanup) `remove --merged` deletes the default-branch worktree itself when the default branch is non-standard

- **Where:** cmd/remove_filter.go + internal/git/protection.go:remove_filter.go:74-79 (CheckMerged), protection.go:5-10
- **Scenario:** User runs `sentei clone <repo>` where the repo's default branch is `production` (the exact case the clone fix now supports). sentei creates a `production` worktree. Later the user runs `sentei remove --merged --non-interactive --force` (or selects 'merged' in the TUI) to clean up merged feature branches.
- **Why broken:** CheckMerged runs `git merge-base --is-ancestor production production`, which returns 0 because a branch is always its own ancestor, so the default worktree is reported as 'merged'. `production` is not in the hardcoded protected set {main,master,develop,dev}, so ResolveFilters does not exclude it. The user's primary working tree is removed. The branch ref survives (recoverable), but the working directory and any uncommitted work in it are destroyed, and the user did not ask to touch their default branch.
- **Fix:** Protection must be data-driven on the ACTUAL default branch, not a hardcoded name list. Detect the real default (via `git symbolic-ref --short HEAD` on the bare repo, same source the clone fix now uses) and always exclude that branch's worktree from removal, in addition to the static set. Also: --merged should never match the default branch against itself (skip when branch == defaultBranch).
- **Confidence:** high | **Verified via:** Full end-to-end live git experiment plus consumer trace. Reproduced the exact sentei clone layout (remote with default branch `production`, bare clone into `.bare`, `.git` pointer, refspec config, `git fetch origin`, `detectDefaultBranch` via `symbolic-ref --short HEAD`, `git worktree add production production`, plus a `feature-merged` worktree). Built the binary and ran the real `sentei remove -merged --non-interactive --force <repo>`. Traced both consumers of ResolveFilters (cmd/remove.go:51 and main.go:223) and the TUI SetRemoveOpts. Confirmed each link of the mechanism with targeted git commands.

## [P0] (gitlayer) protection.go only protects a hardcoded {main,master,develop,dev} set, so a non-standard default branch (trunk/production/mainline) is unprotected and gets force-deleted by `remove --all`

- **Where:** internal/git/protection.go:5-15
- **Scenario:** User has a bare sentei repo whose default branch is `trunk` (set via `git symbolic-ref HEAD refs/heads/trunk`; common in many orgs: trunk/production/mainline/release). They run `sentei remove --all`. `ResolveFilters` excludes bare worktrees and branches where `IsProtectedBranch` returns true, but `IsProtectedBranch("refs/heads/trunk")` is false, so the trunk worktree is included and removed with `git worktree remove --force` (discarding any uncommitted changes). There is no current-working-directory guard anywhere in the remove path (only clone.go calls os.Getwd), so if the command is run from inside the trunk worktree, the user deletes the directory they are standing in. Contrast: the same command with a `main` default correctly protects main.
- **Why broken:** The default branch (and its worktree, often holding the user's primary work) is silently deleted. The tool already KNOWS the real default branch (cmd/remove_filter.go DetectDefaultBranch is used for the --merged ancestor check) but protection ignores that result and relies on a static literal set. This is the precise scenario the fix/detect-default-branch work targets, yet protection was never wired to it.
- **Fix:** Make protection data-driven: have the remove flow compute the detected default branch (DetectDefaultBranch already exists) and pass it into ResolveFilters so it is always excluded, in addition to the static convention set. Root cause is that protection is a context-free static map; the default branch is repo-specific runtime data and must come from `git symbolic-ref --short HEAD` on the bare repo, not a literal list. Also add the missing current-working-directory guard (resolve os.Getwd against each worktree path) so the worktree the user is standing in is never removed.
- **Confidence:** high | **Verified via:** Traced the real consumer (cmd/remove.go::RunRemove -> ResolveFilters -> git worktree remove --force) and the interactive TUI (internal/tui/list.go select-all/toggle), then ran two live end-to-end experiments driving the actual built binary against sentei-layout bare repos whose default branch is `trunk` vs `main`. Also grepped for any CWD guard and for all consumers of IsProtectedBranch / DetectDefaultBranch.

## [P0] (migrate) Slash branch name silently checks out a DIFFERENT new branch; migration reports success

- **Where:** internal/repo/migrate.go:209 (runner.Run(repoPath, "worktree", "add", branch)), 56 (WorktreePath join)
- **Scenario:** User is on a branch containing a slash, e.g. `feature/foo` (extremely common), and runs migrate.
- **Why broken:** `git worktree add feature/foo` treats the argument as a PATH whose basename (`foo`) becomes the new branch name. Git creates and checks out a brand-new branch `foo` in directory `feature/foo`, while the user's real `feature/foo` branch is left orphaned with no worktree. Migration prints '✓ Migration complete' and WorktreePath=<repo>/feature/foo, so it looks correct. The two branches share a commit initially, but every future commit lands on `foo`, silently diverging from `feature/foo`. The user believes they are working on `feature/foo` and is not. This is silent partial success that looks fine but is corrupt.
- **Fix:** Use the explicit form that targets the existing branch and an explicit path: `git worktree add <sanitized-dir-path> <branch>` where the branch is passed as a separate argument so git checks out the EXISTING branch rather than inventing one from the basename. Sanitize the directory name (slug) separately from the branch name; do not conflate path and ref. Match the worktree-creation logic already used in the create/clone flows.
- **Confidence:** high | **Verified via:** Traced the real consumer (cmd/migrate.go gates only on HasFailures) AND ran the built binary end-to-end on a feature/foo checkout, plus isolated git experiments to confirm both the bug and the fix.

## [P0] (migrate) Migrated bare repo's origin URL is rewritten to a dead local path; real remote is permanently severed

- **Where:** internal/repo/migrate.go:137 (clone --bare .git .bare), 150 (rm .git), 173 (sets only remote.origin.fetch)
- **Scenario:** User migrates any repo that has a real remote (e.g. origin = git@github.com:user/proj.git), which is essentially every cloned repo.

Reuse whatever remote-preservation logic clone.go relies on. Mirroring all remotes, not just origin, is preferable.
- **Confidence:** high | **Verified via:** Live git experiment replicating migrate's exact sequence (internal/repo/migrate.go:137 `git clone --bare .git .bare`, os.RemoveAll(.git), write `gitdir: .bare` pointer, set remote.origin.fetch) plus consumer trace of cmd/migrate.go and repo_progress.go, plus grep for any URL-restoration logic.

## [P1] (clone) git fetch origin gates the essential worktree step behind a non-essential tracking fetch (no rollback on network/auth failure)
